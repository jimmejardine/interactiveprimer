Records and Variants

Two type constructors carry almost all of the data in a real program. Records (also called products, structs, or objects) bundle several values together: a point \{x:\mathsf{Int},\, y:\mathsf{Int}\} is an x and a y. Variants (also called sums, tagged unions, or coproducts) offer a value that is one of several alternatives: a shape \langle \mathsf{circle}:\mathsf{Float},\, \mathsf{rect}:\{w,h\}\rangle is a circle or a rectangle. Products are "and", sums are "or", and every algebraic data type in ML, Haskell, Rust or Swift is just a nesting of the two.

Once we have subtyping, records and variants each acquire a rich and — pleasingly — dual theory of when one is safely substitutable for another. Records may drop fields; variants may drop tags; and the direction in which each is allowed to drop is exactly reversed. Getting this duality straight is the key to the whole module, so we will draw it, prove it in code, and then admit where it goes wrong.

Record subtyping: three independent freedoms

A record type is \{l_1:T_1,\dots,l_n:T_n\} — a set of labelled fields. When is one record type a subtype of another? A value of the subtype must be usable wherever the supertype is expected, i.e. it must supply at least the fields the supertype promises, each of a compatible type. That yields three orthogonal rules.

  • Width — a record with more fields is a subtype of one with fewer. Extra fields can always be forgotten: \{l_i:T_i\}_{i\in 1..n+k} \;<:\; \{l_i:T_i\}_{i\in 1..n}.
  • Depth — refine the type of a field, keeping the same labels, and subtyping is preserved field-by-field: \dfrac{S_i <: T_i \ \ (\text{for each } i)}{\{l_i:S_i\}_{i\in 1..n} \;<:\; \{l_i:T_i\}_{i\in 1..n}}.
  • Permutation — field order does not matter; reordering yields mutually-subtype (hence equivalent) types.

These combine: \{x:\mathsf{Nat},\, y:\mathsf{Nat},\, c:\mathsf{Bool}\} <: \{x:\mathsf{Int},\, y:\mathsf{Int}\} by dropping c (width) and refining each remaining field via \mathsf{Nat} <: \mathsf{Int} (depth). Notice the extreme case: the empty record \{\} is a supertype of every record — forget all fields — so \{\} behaves as a \mathsf{Top} for records (and doubles as the unit type).

See width and depth at once

The diagram shows three record types. Moving right forgets and generalises; every leftward record is a subtype of the ones to its right. The first arrow is pure width (drop c); the second is pure depth (\mathsf{Int} refined to \mathsf{Number} in each field, same labels).

The direction is the crux: more information is a subtype of less. A three-field record can masquerade as a two-field one; a two-field record of \mathsf{Int}s can masquerade as one of \mathsf{Number}s. This is precisely how a structurally typed language (TypeScript, Go interfaces, OCaml objects) lets an over-specified object satisfy an under-specified expectation with no explicit "implements" declaration.

Variants are records turned inside out

A variant type \langle l_1:T_1,\dots,l_n:T_n\rangle describes a value tagged with exactly one label l_i and carrying a payload of type T_i. You introduce a variant by injecting one tag, and you eliminate it with a case that must handle every tag:

\dfrac{\Gamma \vdash t : T_j \quad (l_j \in \{l_i\})} {\Gamma \vdash \langle l_j = t\rangle : \langle l_i:T_i\rangle}\;(\textsc{T-Variant}) \qquad \dfrac{\Gamma \vdash t : \langle l_i:T_i\rangle \quad \Gamma, x_i:T_i \vdash u_i : U} {\Gamma \vdash \mathsf{case}\ t\ \mathsf{of}\ \langle l_i = x_i\rangle \Rightarrow u_i : U}\;(\textsc{T-Case})

Now subtyping runs the other way from records. A variant with fewer tags is a subtype of one with more, because a value that can only be a circle can safely flow into a context prepared to handle a circle or a rectangle — the extra case branch simply never fires:

\langle l_i:T_i\rangle_{i\in 1..n} \;<:\; \langle l_i:T_i\rangle_{i\in 1..n+k}\;(\textsc{S-Variant-Width}).

Depth subtyping, by contrast, keeps the same direction as records (covariant in each payload): \langle l:S\rangle <: \langle l:T\rangle when S <: T. So width flips, depth doesn't — that asymmetry is the duality of products and sums.

A structural checker for records and variants

We extend the subtype checker with records and variants. A record is a subtype when it covers every field the supertype requires (width + permutation) with a subtype payload (depth). A variant is a subtype when the supertype covers every tag the subtype could produce — the loop simply quantifies over the other side.

type Ty = | { kind: "prim"; name: string } // Nat | Int | Number | { kind: "rec"; fields: Record<string, Ty> } | { kind: "var"; tags: Record<string, Ty> }; const supers: Record<string, string[]> = { Nat: ["Nat", "Int", "Number"], Int: ["Int", "Number"], Number: ["Number"], }; const primSub = (s: string, t: string) => (supers[s] ?? [s]).includes(t); function subtype(s: Ty, t: Ty): boolean { if (s.kind === "prim" && t.kind === "prim") return primSub(s.name, t.name); if (s.kind === "rec" && t.kind === "rec") { // Width + permutation + depth: s must supply every field t needs, with a subtype. return Object.keys(t.fields).every( (l) => l in s.fields && subtype(s.fields[l], t.fields[l])); } if (s.kind === "var" && t.kind === "var") { // Dual: t must ACCEPT every tag s can produce, covariantly in the payload. return Object.keys(s.tags).every( (l) => l in t.tags && subtype(s.tags[l], t.tags[l])); } return false; } const P = (name: string): Ty => ({ kind: "prim", name }); const R = (fields: Record<string, Ty>): Ty => ({ kind: "rec", fields }); const V = (tags: Record<string, Ty>): Ty => ({ kind: "var", tags }); // Width + depth for records: {x:Nat,y:Nat,c} <: {x:Int,y:Int} console.log("record width+depth:", subtype(R({ x: P("Nat"), y: P("Nat"), c: P("Int") }), R({ x: P("Int"), y: P("Int") }))); // true // ...but never the other way (missing field c, and Int is not <: Nat): console.log("record reversed: ", subtype(R({ x: P("Int"), y: P("Int") }), R({ x: P("Nat"), y: P("Nat"), c: P("Int") }))); // false // Variant WIDTH flips: fewer tags <: more tags. console.log("variant fewer<:more:", subtype(V({ circle: P("Number") }), V({ circle: P("Number"), rect: P("Int") }))); // true console.log("variant more<:fewer:", subtype(V({ circle: P("Number"), rect: P("Int") }), V({ circle: P("Number") }))); // false

The two middle lines expose the duality perfectly: the record check quantifies over the supertype's fields, the variant check over the subtype's tags — mirror images that make records generalise by losing fields and variants generalise by gaining tags.

Count the values. A type with |A| values and one with |B| values combine as a record into |A|\times|B| values (every a paired with every b) and as a variant into |A|+|B| values (an a or a b). That is why they are called products and sums — and the algebra is real: the unit type \mathbf{1}=\{\} (one value) is the identity for products, the empty type \mathbf{0} (zero values, our \mathsf{Bot}) is the identity for sums, and one can even show A\times(B+C)\cong A\times B + A\times C as an isomorphism of types. This "algebra of types" is the origin of the name algebraic data type, and differentiating a type (yes, calculus) yields its zipper — a genuine research result.

Record depth subtyping — refining a field's type — is sound only for fields you can read. The instant a field is mutable, the same trap that sinks covariant arrays reappears. Suppose \{p:\mathsf{Nat}\} <: \{p:\mathsf{Int}\} by depth (\mathsf{Nat} <: \mathsf{Int}). Give a value of the first type to code that sees a \{p:\mathsf{Int}\}; it writes r.p = -5 — a perfectly good \mathsf{Int} — and now a supposed \mathsf{Nat} holds -5. A writable field is invariant; only immutable (readonly) fields may be depth-subtyped. This is exactly why TypeScript treats readonly arrays and properties more permissively than mutable ones, and why sound object calculi split each field into separate read (covariant) and write (contravariant) capabilities.
