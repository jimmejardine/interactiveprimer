The Curry–Howard Correspondence

Here is one of the most astonishing facts in all of computer science, and it hides in plain sight in the derivation trees you have already been drawing. Types are propositions. Programs are proofs. The rules of the simply-typed lambda calculus are, symbol for symbol, the rules of intuitionistic logic — and a typing derivation is literally a formal proof. This is the Curry–Howard correspondence (also the proofs-as-programs or propositions-as-types principle), discovered by Haskell Curry in the 1930s–50s and made precise by William Howard in 1969.

It is not an analogy or a loose metaphor. It is an isomorphism: to every construct on the programming side there is an exactly matching construct on the logic side, and the two run in lock-step. Computing is proving; proving is computing. This single insight underwrites modern proof assistants (Coq, Agda, Lean), dependent types, and the entire idea that a program can be a certificate of its own correctness.

The dictionary

The correspondence is a translation table between two vocabularies for the same objects. Read any row left-to-right to go from logic to programming, or right-to-left to go back.

LogicProgramming (λ→ & extensions)
proposition Atype A
proof of Aprogram (term) of type A
implication A \Rightarrow Bfunction type A \to B
conjunction A \wedge Bproduct / pair type A \times B
disjunction A \vee Bsum / variant type A + B
truth \topunit type \mathbf{1} (one value)
falsity \botempty type \mathbf{0} (no values)
implication-introduction (assume, discharge)\lambda-abstraction
implication-elimination (modus ponens)function application
proof normalisation (cut elimination)program evaluation (\beta-reduction)

The heart of the table is the top block: implication is the function arrow. "If I have a proof of A, I can build a proof of B" is exactly "given a value of type A, I return a value of type B" — a function A \to B. Everything else is elaboration.

λ→ ↔ intuitionistic implicational logic

Restrict to just base types and arrows — the pure \lambda_{\to} — and the correspondence is precise: it is the implicational fragment of intuitionistic propositional logic. Line up the rules and they are the same rules with the terms erased.

\underbrace{\dfrac{\Gamma,\,x{:}A \vdash t : B}{\Gamma \vdash \lambda x{:}A.\,t : A \to B}}_{\textsf{T-Abs}} \;\;\Longleftrightarrow\;\; \underbrace{\dfrac{\Gamma,\,A \vdash B}{\Gamma \vdash A \Rightarrow B}}_{\Rightarrow\text{-Intro}} \underbrace{\dfrac{\Gamma \vdash t_1 : A \to B \qquad \Gamma \vdash t_2 : A}{\Gamma \vdash t_1\,t_2 : B}}_{\textsf{T-App}} \;\;\Longleftrightarrow\;\; \underbrace{\dfrac{\Gamma \vdash A \Rightarrow B \qquad \Gamma \vdash A}{\Gamma \vdash B}}_{\Rightarrow\text{-Elim (modus ponens)}}

Erase the terms from a typing derivation and a natural-deduction proof stares back at you; the context \Gamma of variable bindings is the list of undischarged assumptions. T-Var — "the variable is in the context" — is exactly the logical rule "an assumption in scope may be used". A term is well-typed at A if and only if the proposition A is provable.

  • A type A is inhabited (some closed term \vdash t : A exists) if and only if the proposition A is a theorem of intuitionistic implicational logic.
  • Closed terms of type A correspond bijectively to natural-deduction proofs of A.
  • \beta-reduction of terms corresponds exactly to normalisation (cut-elimination) of proofs.

One tree, two readings

Consider the humble term \lambda x{:}A.\,\lambda y{:}B.\,x — the "constant function" that takes two arguments and returns the first. Its type is A \to B \to A. Under Curry–Howard, that type is the proposition A \Rightarrow (B \Rightarrow A) — a tautology ("if A, then B implies A", i.e. a true thing stays true under any extra assumption). The leaf uses the assumption x{:}A — logically, "having assumed A (and B), A holds". Each \lambda that wraps the body is a \Rightarrow-introduction that discharges an assumption: the inner \lambda y{:}B discharges B to prove B \Rightarrow A; the outer \lambda x{:}A discharges A to prove the closed theorem A \Rightarrow (B \Rightarrow A). Building the program is writing the proof.

Combinators are logical axioms

The correspondence has a delightful concrete face: the classic combinators \mathbf{K} and \mathbf{S} have types that are exactly the two axiom schemes of Hilbert-style implicational logic:

\mathbf{K} : A \to B \to A \qquad\qquad \mathbf{S} : (A \to B \to C) \to (A \to B) \to A \to C

As propositions these read A \Rightarrow (B \Rightarrow A) and (A \Rightarrow (B \Rightarrow C)) \Rightarrow ((A \Rightarrow B) \Rightarrow (A \Rightarrow C)) — the two axioms from which all of implicational logic is derived by modus ponens (application). The identity \mathbf{I} = \mathbf{S}\,\mathbf{K}\,\mathbf{K} proves A \Rightarrow A. TypeScript's generics let us write and run these very proofs:

// K : A → B → A — the proof of A ⇒ (B ⇒ A). const K = <A, B>(a: A) => (_b: B): A => a; // S : (A→B→C) → (A→B) → A → C — the proof of the distribution axiom. const S = <A, B, C>(f: (a: A) => (b: B) => C) => (g: (a: A) => B) => (a: A): C => f(a)(g(a)); // I = S K K : A → A — the proof of A ⇒ A, BUILT from the axioms by modus ponens (application). const I = <A>(x: A): A => S<A, () => A, A>(K)(K)(x); console.log("K picks the first arg :", K("keep")("drop")); // "keep" console.log("I is the identity :", I(42)); // 42 console.log("S applied (f a)(g a) :", S((n: number) => (m: number) => n + m)((n: number) => n * 10)(3)); // 3 + 30 = 33 // Conjunction A∧B is a PAIR; the projections are ∧-elimination. const pair = <A, B>(a: A, b: B): [A, B] => [a, b]; // ∧-introduction const fst = <A, B>(p: [A, B]): A => p[0]; // A ∧ B ⇒ A const snd = <A, B>(p: [A, B]): B => p[1]; // A ∧ B ⇒ B const p = pair("proofOfA", 7); console.log("∧-intro then ∧-elim (fst):", fst(p), "| (snd):", snd(p));

Products, sums, and the empty type

Extend \lambda_{\to} with pairs and variants and the logic grows to match exactly. A product A \times B is a conjunction: to hold one you must hold both A and B, and its projections \pi_1, \pi_2 are the \wedge-elimination rules. A sum A + B is a disjunction: to hold one you supply either an A or a B (tagged Left/Right), and case-analysis is \vee-elimination. The unit type \mathbf{1}, with its single trivial value, is \top (trivially true). And the empty type \mathbf{0}, with no values, is \bot (falsity) — the fact that you can never construct a value of it mirrors the fact that you can never prove a contradiction. Negation \neg A is then just A \to \mathbf{0}: a function turning any proof of A into the impossible.

The correspondence was hiding in the notation the whole time. Gentzen's natural deduction (1934) and Church's typed lambda calculus (1940) were built by different communities — logicians and computability theorists — who rarely read each other's papers. Curry noticed in the 1950s that the types of his combinators matched Hilbert's axioms, but it took William Howard's privately-circulated 1969 note (finally published in 1980, in a festschrift for Curry) to spell out the full rule-for-rule match, including the reduction/normalisation half. The idea was rediscovered so many times — by de Bruijn, by Läuchli, by Martin-Löf — that it is sometimes called the Curry–Howard–de Bruijn–Lambek correspondence. Its modern reach is enormous: when you write a program in Coq or Lean and the type-checker accepts it, you have proved a theorem, and the program is the proof object. Mathematics and programming turn out to be the same activity wearing two hats.

λ→ corresponds to INTUITIONISTIC logic, not classical logic. The difference is not a technicality — some classical tautologies are simply not inhabited. There is no closed term of type ((A \to \mathbf{0}) \to \mathbf{0}) \to A (double-negation elimination) or of type A + (A \to \mathbf{0}) (the law of the excluded middle, A \vee \neg A). Constructively, to prove A \vee \neg A you must actually produce either a proof of A or a refutation — and no uniform program can do that for an arbitrary A. Recovering classical logic requires adding a control operator like call/cc, whose type is Peirce's law ((A \to B) \to A) \to A — Griffin's 1990 discovery that control flow is classical reasoning. So "types are propositions" is true, but which logic you get depends on which features your language has.
